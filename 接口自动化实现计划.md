# CaseFlow 接口自动化测试实现计划

## 一、项目概述

### 1.1 当前系统架构

| 层次 | 技术栈 |
|------|--------|
| 后端 | Spring Boot 3.2.5 + Java 17 + MyBatis-Plus 3.5.6 |
| 认证 | Sa-Token 1.44.0 (Redis 存储 Session) |
| 数据库 | MySQL |
| 前端 | Vue 3 + Vite + TypeScript |
| UI 组件 | Ant Design Vue + Element Plus |
| 通信 | REST API + WebSocket |
| 文件 | Apache POI (Excel 导入/导出) |

### 1.2 目标

在现有 CaseFlow 测试用例管理系统中，新增 **接口自动化测试** 模块，允许用户：
- 定义 HTTP 接口（URL、Method、Header、Body、鉴权、标签等）
- 为每个接口创建多个用例（不同的请求数据 + 断言组合）
- 将多个用例编排为测试场景，支持变量传递和前后置脚本
- 创建自动化测试计划，选择多个场景批量执行
- 查看执行报告，实时 WebSocket 推送执行进度

## 二、核心概念与流程

### 2.1 四层架构模型

```
┌────────────────────────────────────────────────────────────────┐
│                      自动化测试计划                              │
│           选择多个「测试场景」，批量执行，生成汇总报告              │
├────────────────────────────────────────────────────────────────┤
│                        测试场景                                 │
│         编排多个「接口用例」，按顺序执行，支持变量传递              │
│         可设置前置/后置脚本，步骤间延迟，失败策略                  │
├────────────────────────────────────────────────────────────────┤
│                        接口用例                                 │
│        同一接口的不同测试数据 + 断言规则组合                      │
│        如：正常登录、密码错误、账号锁定、参数缺失 ...              │
├────────────────────────────────────────────────────────────────┤
│                        接口定义                                 │
│        HTTP 接口基础信息：Method、URL、默认Header、鉴权、标签      │
│        属于某个项目 → 某个目录                                   │
└────────────────────────────────────────────────────────────────┘
```

### 2.2 完整操作流程

```
① 选择项目
    │
    ▼
② 在项目下管理目录（复用现有目录树，dir_type = 'API'）
    │
    ▼
③ 在目录下创建「接口定义」
   ┌─────────────────────────────────────┐
   │ 名称: 用户登录                        │
   │ Method: POST                         │
   │ Path: /api/auth/login                │
   │ 鉴权: 无需鉴权                        │
   │ 标签: [用户模块, 核心接口, P0]         │
   │ 默认 Headers / Body 模板              │
   └─────────────────────────────────────┘
    │
    ▼
④ 为该接口创建多个「接口用例」
   ┌──────────────────────────────────────────────┐
   │ 用例1: 正常登录                                │
   │   Body: {"username":"admin","password":"123"}  │
   │   断言: status=200, $.data.token EXISTS        │
   ├──────────────────────────────────────────────┤
   │ 用例2: 密码错误                                │
   │   Body: {"username":"admin","password":"xxx"}  │
   │   断言: status=200, $.code=401                 │
   ├──────────────────────────────────────────────┤
   │ 用例3: 参数缺失                                │
   │   Body: {"username":"admin"}                   │
   │   断言: status=400                              │
   └──────────────────────────────────────────────┘
    │
    ▼
⑤ 创建「测试场景」，选择多个用例编排执行顺序
   ┌──────────────────────────────────────────────┐
   │ 场景: 用户登录 → 获取列表 → 创建用户            │
   │   Step 1: 登录接口 - 正常登录用例               │
   │     → 后置脚本: 提取 $.data.token → {{token}}  │
   │   Step 2: 用户列表接口 - 正常查询用例            │
   │     → 前置脚本: Header Authorization={{token}} │
   │   Step 3: 创建用户接口 - 正常创建用例            │
   │     → 前置脚本: Header Authorization={{token}} │
   └──────────────────────────────────────────────┘
    │
    ▼
⑥ 创建「自动化测试计划」，选择多个场景
   ┌──────────────────────────────────────────────┐
   │ 计划: v2.0 回归测试                             │
   │   环境: 测试环境                                │
   │   场景1: 用户模块 CRUD 流程                      │
   │   场景2: 项目管理 CRUD 流程                      │
   │   场景3: 用例集导入导出流程                       │
   │   → 执行 → 生成报告                             │
   └──────────────────────────────────────────────┘
```

### 2.3 设计合理性分析

| 优势 | 说明 |
|------|------|
| **关注点分离** | 接口定义只关心"是什么接口"，用例关心"怎么测"，场景关心"测试流程"，计划关心"测什么" |
| **高复用性** | 同一个接口用例可以被多个场景引用，同一个场景可以被多个计划引用 |
| **灵活组合** | 可以单独执行用例调试、执行场景验证流程、执行计划做回归测试 |
| **数据隔离** | 接口定义修改不影响已有用例（用例保存自己的数据副本） |
| **标签筛选** | 接口和用例都支持标签，可按标签快速筛选和组合 |

## 三、技术选型

### 3.1 核心技术方案对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **方案A：纯后端 Java 实现** | 与现有系统完全集成，无额外依赖 | 需自行实现 HTTP 客户端、断言引擎 | ⭐⭐⭐⭐ |
| **方案B：集成 RestAssured** | 成熟的 Java HTTP 测试库，DSL 优雅 | 主要面向单元测试，非在线执行 | ⭐⭐⭐ |
| **方案C：集成 HttpRunner** | Python 生态丰富，功能完善 | 需跨语言调用，架构复杂 | ⭐⭐ |

### 3.2 推荐方案：方案 A — 纯后端 Java 实现

**核心依赖：**

```xml
<!-- HTTP 客户端 -->
<dependency>
    <groupId>com.squareup.okhttp3</groupId>
    <artifactId>okhttp</artifactId>
    <version>4.12.0</version>
</dependency>

<!-- JSON 断言 (JSONPath) -->
<dependency>
    <groupId>com.jayway.jsonpath</groupId>
    <artifactId>json-path</artifactId>
    <version>2.9.0</version>
</dependency>

<!-- 表达式引擎 (变量替换、动态取值) -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
```

**选择理由：**
- OkHttp：轻量高效，支持同步/异步、连接池、拦截器
- JsonPath：强大的 JSON 响应提取和断言能力
- 完全在 Java 生态内，与现有 Spring Boot 系统无缝集成
- 无需额外部署 Python 或其他服务

## 四、数据库设计

### 4.1 新增表结构

```sql
-- ========================================
-- 接口自动化：环境配置
-- ========================================
CREATE TABLE api_environments (
    id              VARCHAR(32)  NOT NULL,
    project_id      VARCHAR(32)  NOT NULL COMMENT '所属项目',
    name            VARCHAR(100) NOT NULL COMMENT '环境名称(开发/测试/生产)',
    base_url        VARCHAR(500) NOT NULL COMMENT '基础URL, 如 http://192.168.1.100:8080',
    headers         JSON         DEFAULT NULL COMMENT '全局请求头 {"Content-Type":"application/json"}',
    variables       JSON         DEFAULT NULL COMMENT '环境变量 {"token":"xxx","userId":"123"}',
    created_by      VARCHAR(32),
    created_by_name VARCHAR(100),
    created_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    INDEX idx_project (project_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='接口自动化-环境配置';

-- ========================================
-- 接口自动化：接口定义（基础信息）
-- ========================================
CREATE TABLE api_definitions (
    id              VARCHAR(32)  NOT NULL,
    project_id      VARCHAR(32)  NOT NULL COMMENT '所属项目',
    directory_id    VARCHAR(32)  DEFAULT NULL COMMENT '所属目录(复用directories表, dir_type=API)',
    name            VARCHAR(200) NOT NULL COMMENT '接口名称',
    method          VARCHAR(10)  NOT NULL DEFAULT 'GET' COMMENT 'HTTP方法: GET/POST/PUT/DELETE/PATCH',
    path            VARCHAR(500) NOT NULL COMMENT '请求路径(相对baseUrl), 如 /api/auth/login',
    description     TEXT         DEFAULT NULL COMMENT '接口描述/备注',
    -- 鉴权配置
    auth_type       VARCHAR(30)  DEFAULT 'NONE' COMMENT '鉴权方式: NONE/BEARER_TOKEN/BASIC/API_KEY/CUSTOM',
    auth_config     JSON         DEFAULT NULL COMMENT '鉴权配置详情, 根据auth_type不同而不同',
    -- 默认请求模板（用例可继承或覆盖）
    default_headers JSON         DEFAULT NULL COMMENT '默认请求头 JSON [{"key":"","value":"","desc":""}]',
    default_params  JSON         DEFAULT NULL COMMENT '默认Query参数 JSON [{"key":"","value":"","desc":""}]',
    default_body_type VARCHAR(20) DEFAULT 'NONE' COMMENT 'NONE/JSON/FORM/FORM_DATA/RAW/XML',
    default_body    TEXT         DEFAULT NULL COMMENT '默认请求体内容(模板)',
    -- 标签
    tags            JSON         DEFAULT NULL COMMENT '标签数组 ["用户模块","P0","核心接口"]',
    sort_order      INT          NOT NULL DEFAULT 0,
    created_by      VARCHAR(32),
    created_by_name VARCHAR(100),
    created_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    INDEX idx_project (project_id),
    INDEX idx_directory (directory_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='接口自动化-接口定义';

-- ========================================
-- 接口自动化：接口用例（同一接口的不同测试数据）
-- ========================================
CREATE TABLE api_cases (
    id              VARCHAR(32)  NOT NULL,
    api_id          VARCHAR(32)  NOT NULL COMMENT '所属接口定义',
    name            VARCHAR(200) NOT NULL COMMENT '用例名称, 如: 正常登录 / 密码错误 / 参数缺失',
    description     TEXT         DEFAULT NULL COMMENT '用例说明',
    -- 请求数据（覆盖接口定义的默认值）
    headers         JSON         DEFAULT NULL COMMENT '请求头(覆盖或追加)',
    query_params    JSON         DEFAULT NULL COMMENT 'Query参数(覆盖或追加)',
    body_type       VARCHAR(20)  DEFAULT NULL COMMENT '请求体类型(NULL则继承接口定义)',
    body_content    TEXT         DEFAULT NULL COMMENT '请求体内容',
    -- 前置/后置脚本
    pre_script      JSON         DEFAULT NULL COMMENT '前置脚本(设置变量/Header等)',
    post_script     JSON         DEFAULT NULL COMMENT '后置脚本(提取响应变量等)',
    -- 标签和优先级
    tags            JSON         DEFAULT NULL COMMENT '用例标签 ["冒烟","正向"]',
    priority        VARCHAR(10)  DEFAULT 'P1' COMMENT '优先级: P0/P1/P2/P3',
    enabled         TINYINT      NOT NULL DEFAULT 1 COMMENT '是否启用',
    sort_order      INT          NOT NULL DEFAULT 0,
    created_by      VARCHAR(32),
    created_by_name VARCHAR(100),
    created_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    INDEX idx_api (api_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='接口自动化-接口用例';

-- ========================================
-- 接口自动化：断言规则（关联到用例）
-- ========================================
CREATE TABLE api_assertions (
    id              VARCHAR(32)  NOT NULL,
    case_id         VARCHAR(32)  NOT NULL COMMENT '所属用例',
    type            VARCHAR(30)  NOT NULL COMMENT 'STATUS_CODE/HEADER/JSON_PATH/JSON_SCHEMA/BODY_CONTAINS/RESPONSE_TIME',
    expression      VARCHAR(500) DEFAULT NULL COMMENT 'JSONPath表达式 或 Header名',
    operator        VARCHAR(20)  NOT NULL COMMENT 'EQUALS/NOT_EQUALS/CONTAINS/GT/LT/GTE/LTE/EXISTS/NOT_EXISTS/REGEX',
    expected_value  VARCHAR(1000) DEFAULT NULL COMMENT '期望值',
    sort_order      INT          NOT NULL DEFAULT 0,
    PRIMARY KEY (id),
    INDEX idx_case (case_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='接口自动化-断言规则';

-- ========================================
-- 接口自动化：测试场景（编排多个用例）
-- ========================================
CREATE TABLE api_scenarios (
    id              VARCHAR(32)  NOT NULL,
    project_id      VARCHAR(32)  NOT NULL,
    name            VARCHAR(200) NOT NULL COMMENT '场景名称, 如: 用户CRUD完整流程',
    description     TEXT         DEFAULT NULL,
    -- 场景级配置
    fail_strategy   VARCHAR(20)  DEFAULT 'STOP' COMMENT '失败策略: STOP=遇到失败停止 / CONTINUE=继续执行',
    timeout_ms      INT          DEFAULT 300000 COMMENT '场景总超时(毫秒), 默认5分钟',
    tags            JSON         DEFAULT NULL COMMENT '场景标签',
    created_by      VARCHAR(32),
    created_by_name VARCHAR(100),
    created_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    INDEX idx_project (project_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='接口自动化-测试场景';

-- ========================================
-- 接口自动化：场景步骤（场景中的每个用例）
-- ========================================
CREATE TABLE api_scenario_steps (
    id              VARCHAR(32)  NOT NULL,
    scenario_id     VARCHAR(32)  NOT NULL COMMENT '所属场景',
    case_id         VARCHAR(32)  NOT NULL COMMENT '关联的接口用例',
    sort_order      INT          NOT NULL DEFAULT 0 COMMENT '执行顺序',
    -- 步骤级覆盖（可选，优先级高于用例本身的数据）
    override_headers JSON        DEFAULT NULL COMMENT '覆盖请求头',
    override_body   TEXT         DEFAULT NULL COMMENT '覆盖请求体',
    pre_script      JSON         DEFAULT NULL COMMENT '步骤级前置脚本(追加)',
    post_script     JSON         DEFAULT NULL COMMENT '步骤级后置脚本(追加)',
    delay_ms        INT          DEFAULT 0 COMMENT '执行前延迟(毫秒)',
    retry_count     INT          DEFAULT 0 COMMENT '失败重试次数',
    enabled         TINYINT      NOT NULL DEFAULT 1 COMMENT '是否启用',
    PRIMARY KEY (id),
    INDEX idx_scenario (scenario_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='接口自动化-场景步骤';

-- ========================================
-- 接口自动化：自动化测试计划
-- ========================================
CREATE TABLE api_test_plans (

    description      TEXT         DEFAULT NULL,
    environment_id   VARCHAR(32)  NOT NULL COMMENT '执行环境',
    -- 执行配置
    parallel         TINYINT      DEFAULT 0 COMMENT '场景是否并行执行: 0=串行 1=并行',
    cron_expression  VARCHAR(100) DEFAULT NULL COMMENT '定时执行Cron表达式(为空则手动执行)',
    status           VARCHAR(20)  DEFAULT 'DRAFT' COMMENT 'DRAFT/SCHEDULED/RUNNING/COMPLETED',
    -- 审计字段
    created_by       VARCHAR(32),
    created_by_name  VARCHAR(100),
    updated_by       VARCHAR(32),
    updated_by_name  VARCHAR(100),
    created_at       DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at       DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id),

-- 接口自动化：测试计划关联的场景

-- 接口自动化：执行记录（计划/场景/单用例均可触发）

    failed_cases    INT          DEFAULT 0,
    error_cases     INT          DEFAULT 0,
    skipped_cases   INT          DEFAULT 0,
    duration_ms     BIGINT       DEFAULT 0 COMMENT '总耗时(毫秒)',

    executed_by_name VARCHAR(100),
    started_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    finished_at     DATETIME     DEFAULT NULL,
    PRIMARY KEY (id),
    INDEX idx_project (project_id),
    INDEX idx_plan (plan_id),
    INDEX idx_scenario (scenario_id)

-- 接口自动化：执行详情（每个用例的执行结果）
-- ========================================
CREATE TABLE api_execution_details (
    id                VARCHAR(32)  NOT NULL,
    execution_id      VARCHAR(32)  NOT NULL COMMENT '所属执行记录',
    scenario_id       VARCHAR(32)  DEFAULT NULL COMMENT '所属场景',
    case_id           VARCHAR(32)  NOT NULL COMMENT '执行的用例',
    api_id            VARCHAR(32)  NOT NULL COMMENT '对应的接口定义',
    step_order        INT          NOT NULL DEFAULT 0 COMMENT '在场景中的顺序',
    -- 请求快照
    request_url       VARCHAR(1000),
    request_method    VARCHAR(10),
    request_headers   JSON,
    request_body      TEXT,
    -- 响应快照
    response_status   INT          DEFAULT NULL COMMENT 'HTTP状态码',
    response_headers  JSON,
    response_body     LONGTEXT     DEFAULT NULL,
    -- 结果
    duration_ms       BIGINT       DEFAULT 0 COMMENT '单步耗时',
    assertion_results JSON         DEFAULT NULL COMMENT '[{"type":"","expression":"","pass":true,"actual":"","expected":""}]',
    extracted_vars    JSON         DEFAULT NULL COMMENT '本步提取的变量 {"token":"xxx"}',

    error_message     TEXT         DEFAULT NULL COMMENT '错误信息(异常时)',

    PRIMARY KEY (id),
    INDEX idx_execution (execution_id),
    INDEX idx_case (case_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='接口自动化-执行详情';
```

### 4.3 鉴权配置详解

`api_definitions.auth_type` 和 `auth_config` 的对应关系：

| auth_type | auth_config 示例 | 说明 |
|-----------|------------------|------|
| `NONE` | `null` | 无需鉴权 |
| `BEARER_TOKEN` | `{"token": "{{token}}"}` | Bearer Token，支持变量引用 |
| `BASIC` | `{"username": "admin", "password": "{{pwd}}"}` | HTTP Basic Auth |
| `API_KEY` | `{"key": "X-API-Key", "value": "{{apiKey}}", "in": "HEADER"}` | API Key，可放在 Header 或 Query |
| `CUSTOM` | `{"headers": {"Authorization": "Custom {{sign}}"}}` | 自定义鉴权头 |

鉴权配置会在执行时自动注入到请求头中，优先级：**用例覆盖 > 接口定义 > 环境全局Header**。

#### 5.2.1 HTTP 执行器 (`HttpExecutor`)

```java
@Component
public class HttpExecutor {
    private final OkHttpClient client;

    public HttpResult execute(ExecuteRequest request) {
        // 1. 构建完整 URL = baseUrl + path + queryParams

        // 6. 读取响应，封装 HttpResult

public record HttpResult(
```

#### 5.2.2 鉴权注入器 (`AuthInjector`)

```java
@Component
public class AuthInjector {
    public void inject(RequestBuilder builder, String authType, Map<String, Object> authConfig,
                       VariableContext vars) {
        switch (authType) {
            case "BEARER_TOKEN" -> {
                String token = vars.resolve(authConfig.get("token").toString());
                builder.addHeader("Authorization", "Bearer " + token);
            }
            case "BASIC" -> {
                String user = vars.resolve(authConfig.get("username").toString());
                String pass = vars.resolve(authConfig.get("password").toString());
                builder.addHeader("Authorization", "Basic " + Base64.encode(user + ":" + pass));
            }
            case "API_KEY" -> {
                String key = authConfig.get("key").toString();
                String value = vars.resolve(authConfig.get("value").toString());
                String in = authConfig.getOrDefault("in", "HEADER").toString();
                if ("HEADER".equals(in)) builder.addHeader(key, value);
                else builder.addQueryParam(key, value);
            }
            case "CUSTOM" -> {
                // 自定义 Header 注入
            }
        }
    }
}
```

#### 5.2.3 变量替换引擎 (`VariableResolver`)

```java
@Component
public class VariableResolver {
    private static final Pattern VAR_PATTERN = Pattern.compile("\\{\\{(.+?)\\}\\}");

    /**
     * 变量来源优先级（高 → 低）:
     * 1. 上一步提取的变量（post_script extract）
     * 2. 场景级运行时变量
     * 3. 环境变量 (api_environments.variables)
     * 4. 内置函数变量 ($timestamp, $uuid, $randomInt 等)
     */
    public String resolve(String template, VariableContext context) {
        if (template == null) return null;
        return VAR_PATTERN.matcher(template).replaceAll(m -> {
            String key = m.group(1).trim();
            if (key.startsWith("$")) return evalBuiltin(key);
            return context.getOrDefault(key, m.group(0));
        });
    }
}
```

**变量语法：**

| 语法 | 说明 | 示例 |
|------|------|------|
| `{{varName}}` | 普通变量替换 | `{{baseUrl}}/api/login` |
| `{{$randomInt}}` | 随机整数 (1-999999) | `user_{{$randomInt}}` |
| `{{$randomInt(1,100)}}` | 指定范围随机整数 | `age={{$randomInt(18,60)}}` |
| `{{$timestamp}}` | 当前时间戳(毫秒) | `ts={{$timestamp}}` |
| `{{$timestampSec}}` | 当前时间戳(秒) | `ts={{$timestampSec}}` |
| `{{$uuid}}` | UUID | `id={{$uuid}}` |
| `{{$randomEmail}}` | 随机邮箱 | `email={{$randomEmail}}` |
| `{{$randomStr(10)}}` | 随机字符串(指定长度) | `name={{$randomStr(8)}}` |
| `{{$now(yyyy-MM-dd)}}` | 当前时间(指定格式) | `date={{$now(yyyy-MM-dd)}}` |

#### 5.2.4 断言引擎 (`AssertionEngine`)

```java
@Component
public class AssertionEngine {
    public AssertionResult evaluate(ApiAssertion assertion, HttpResult result) {
        String actual = extractActual(assertion, result);
        boolean pass = compare(actual, assertion.getOperator(), assertion.getExpectedValue());
        return new AssertionResult(assertion, pass, actual);
    }

    private String extractActual(ApiAssertion assertion, HttpResult result) {
        return switch (assertion.getType()) {
            case "STATUS_CODE"    -> String.valueOf(result.statusCode());
            case "JSON_PATH"      -> JsonPath.read(result.body(), assertion.getExpression()).toString();
            case "HEADER"         -> result.headers().get(assertion.getExpression());
            case "BODY_CONTAINS"  -> result.body();
            case "RESPONSE_TIME"  -> String.valueOf(result.durationMs());
            default -> throw new IllegalArgumentException("Unknown type: " + assertion.getType());
        };
    }
}
```

**支持的断言类型：**

| 类型 | 说明 | 示例 |
|------|------|------|
| `STATUS_CODE` | HTTP 状态码 | 等于 200 |
| `JSON_PATH` | JSON 响应字段值 | `$.data.id` 不为空 |
| `JSON_SCHEMA` | JSON 结构校验 | 响应符合 Schema |
| `HEADER` | 响应头值 | `Content-Type` 包含 `json` |
| `BODY_CONTAINS` | 响应体包含文本 | 包含 `"success"` |
| `RESPONSE_TIME` | 响应时间(ms) | 小于 3000ms |

**支持的操作符：**

| 操作符 | 说明 | 示例 |
|--------|------|------|
| `EQUALS` | 等于 | 状态码 = 200 |
| `NOT_EQUALS` | 不等于 | code != 500 |
| `CONTAINS` | 包含 | body 包含 "success" |
| `NOT_CONTAINS` | 不包含 | body 不包含 "error" |
| `GT` | 大于 | total > 0 |
| `LT` | 小于 | response_time < 3000 |
| `GTE` | 大于等于 | count >= 1 |
| `LTE` | 小于等于 | duration <= 5000 |
| `EXISTS` | 存在(非null) | $.data.token EXISTS |
| `NOT_EXISTS` | 不存在 | $.error NOT_EXISTS |
| `REGEX` | 正则匹配 | email 匹配 `^.+@.+$` |
| `IS_EMPTY` | 为空 | $.data.list IS_EMPTY |
| `IS_NOT_EMPTY` | 非空 | $.data.records IS_NOT_EMPTY |

## 十、技术要点和注意事项

### 10.1 安全性

- 所有接口请求通过后端代理发送（避免浏览器 CORS 限制）
- 环境变量中的敏感信息（密码、Token）支持加密存储
- 执行日志脱敏处理（隐藏 Authorization 头中的完整 Token）
- 限制并发执行数量（线程池 maxPoolSize=20），防止资源耗尽
- 请求目标 URL 白名单/黑名单（防止 SSRF 攻击内网）

### 10.2 性能

- HTTP 连接池复用（OkHttp ConnectionPool, maxIdleConnections=10）
- 异步执行避免阻塞主线程
- 响应体过大时截断存储（限制 1MB，超出部分保存摘要）
- 执行记录分页查询，支持按时间范围筛选
- 定期清理过期执行记录（保留 N 天，SysJob 定时清理）

### 10.3 可靠性

- 执行过程中单步异常不影响后续步骤（可配置：STOP / CONTINUE）
- 超时控制：单步超时默认 30s，场景总超时默认 5min
- 重试机制：步骤级可配置重试次数
- 执行取消：支持中途取消正在执行的计划
- 异常恢复：服务重启后，标记 RUNNING 状态为 ERROR

### 10.4 可扩展性

- 断言类型支持插件化扩展（实现 AssertionEvaluator 接口）
- 鉴权方式支持扩展（实现 AuthProvider 接口）
- 脚本引擎预留 Groovy/JavaScript 执行能力
- 支持后续扩展为分布式执行（多 Worker 节点并行）

## 十二、总结

### 核心架构：四层模型

```
接口定义 (API Definition)
    │  1:N
接口用例 (API Case)         ← 同一接口的不同测试数据/断言组合
    │  N:N (通过场景步骤)
测试场景 (API Scenario)     ← 编排多个用例，变量传递，流程验证
    │  N:N (通过计划关联)
测试计划 (API Test Plan)    ← 批量执行多个场景，生成汇总报告
```

### 设计优势

1. **四层分离**：接口定义 → 用例 → 场景 → 计划，各层职责清晰
2. **高度复用**：一个接口多个用例，一个用例可被多个场景引用，一个场景可被多个计划引用
3. **灵活执行**：支持单用例调试、单场景执行、计划批量执行三种粒度
4. **鉴权完整**：支持 NONE/BEARER/BASIC/API_KEY/CUSTOM 五种鉴权方式
5. **标签体系**：接口、用例、场景均支持标签，便于筛选和组织
6. **权限细粒度**：20+ 权限码，完全复用现有 RBAC 系统
7. **纯 Java 实现**：与现有 Spring Boot 系统无缝集成
8. **复用现有基础设施**：目录管理、权限系统、通知系统、WebSocket
9. **渐进式交付**：分 4 阶段实施，每阶段可独立使用

### 预计总工期

**6~10 周**（根据开发人员数量和熟练度浮动）

| 阶段 | 周期 | 交付物 |
|------|------|--------|
| 第一阶段 | 1.5~2 周 | 环境管理 + 接口定义 CRUD + 目录 + 权限 |
| 第二阶段 | 2~3 周 | 用例管理 + 调试执行 + 断言 + 变量 + 鉴权 |
| 第三阶段 | 2~3 周 | 场景编排 + 计划管理 + 异步执行 + 报告 |
| 第四阶段 | 1~2 周 | 导入导出 + 定时执行 + CI/CD + 数据驱动 |
